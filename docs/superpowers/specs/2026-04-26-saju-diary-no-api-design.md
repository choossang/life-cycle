# 사주 4행일기 보강 설계서 (API 미사용 / 보안 우선)

작성일: 2026-04-26
범위: 구현 전 설계 문서화(코드 변경 없음)

## 1) 한 줄 요약(아주 쉽게)

- 점수판(사주 점수)은 그대로 두고,
- 일기장은 설명 메모로만 쓰며,
- 저장/삭제는 버튼(form POST)으로만 처리한다.

즉, **"점수는 고정, 일기는 메모, 서버가 직접 처리"** 구조다.

---

## 2) 확정 원칙

1. 회고/당월 일기는 동일 가중치로 취급한다.
2. `job/gui/hlt` 사주 점수는 원본 불변이다.
3. 일기는 점수를 바꾸지 않고 해석 근거로만 사용한다.
4. API(fetch/json) 없이 서버 렌더 + HTML form submit 방식으로 동작한다.
5. 합법적 사용/개인정보 보호/보안 우선 원칙을 지킨다.

---

## 3) 현재 코드 위험(왜 바꿔야 하는가)

기존 구조는 `/api/saju`, `/api/diary`를 JS `fetch()`로 호출하고, 일부 렌더를 `innerHTML`로 구성한다.

주요 위험:
- 저장형 XSS 위험 (`innerHTML` 기반 문자열 삽입)
- 세션 기반 상태 변경 요청의 CSRF 방어 부족
- 입력 형식/길이 검증 약함

따라서 **API를 걷어내고 서버에서 안전하게 렌더링**하는 구조로 단순화한다.

---

## 4) 목표 아키텍처 (No-API SSR)

### 라우트 구성

1. `GET /`
- 인증 확인
- 사주 월별 데이터 + 선택 월 + 해당 월 일기 데이터를 서버가 조회
- Jinja 템플릿으로 HTML 렌더링

2. `POST /diary/save`
- CSRF 검증
- 입력 검증(`date`, `f1~f4`, 메타 필드)
- DB 저장(upsert)
- `redirect('/?date=YYYY-MM')` (PRG 패턴)

3. `POST /diary/delete`
- CSRF 검증
- 삭제 처리
- `redirect('/?date=YYYY-MM')`

### 데이터 흐름

- 브라우저는 페이지를 받고(form 포함), 버튼 클릭 시 폼 제출만 한다.
- 서버가 검증/저장/삭제를 수행하고 다시 화면을 그려준다.
- 브라우저 JS는 필수 로직이 아니라 보조 UI에만 제한한다.

---

## 5) 데이터 모델 설계

기존 `diary(date, f1, f2, f3, f4, updated)`를 다음으로 확장한다.

- `entry_type` TEXT (`manual` | `retro`)
- `confidence` REAL (0.0~1.0, 기본 1.0)
- `source_note` TEXT (선택)
- `created_at` TEXT
- `updated` TEXT

무결성 규칙:
- 저장용 date 포맷은 내부적으로 `YYYY-MM` 단일 표준
- 화면 표시는 `YYYY년 MM월`로 변환 가능
- f1~f4는 길이 제한(예: 1000자)

---

## 6) 보안 설계 (필수)

1. XSS 차단
- 사용자 입력을 `innerHTML`로 삽입하지 않는다.
- 템플릿 escape를 기본값으로 사용한다.

2. CSRF 차단
- 모든 상태 변경 form(`save/delete`)에 CSRF 토큰 포함
- 서버에서 토큰 검증 실패 시 403

3. 입력 검증
- date 포맷 검증
- f1~f4 타입/길이 검증
- `entry_type/confidence` 범위 검증

4. 세션/쿠키 보안
- `SESSION_COOKIE_HTTPONLY=True`
- `SESSION_COOKIE_SAMESITE='Lax'` 이상
- HTTPS에서 `SESSION_COOKIE_SECURE=True`

5. 비밀정보/개인정보
- `SECRET_KEY`, `APP_PASSWORD`, `DB_PATH`는 Render 환경변수만 사용
- `.env`, `diary.db`는 git 커밋 금지

---

## 7) 정합성 규칙 (사주 기반 우선)

- 점수 계산 입력은 오직 사주 데이터(`saju_data.json`)로 제한
- 일기는 점수 계산에 참여하지 않고 해석 문구/근거 표시에만 사용
- 회고(`retro`)와 당월(`manual`)은 같은 무게로 저장하되, 출처 메타를 남긴다
- 테스트로 "일기 입력 전후 점수 동일"을 항상 검증한다

## 8) Render 무료 플랜 기준 운영 전략

- 구조 단순화(SSR + form)로 운영/디버깅 부담 최소화
- DB는 유실 방지 전략 필요:
  - 우선순위 1: Render Persistent Disk 경로를 `DB_PATH`로 지정
  - 대안: 외부 DB 사용
- 배포 후 스모크 테스트:
  - 로그인
  - 월 선택/표시
  - 일기 저장/수정/삭제
  - 재시작 후 데이터 유지 확인

## 10) Stitch 연계 원칙

Stitch에서 디자인을 보완해도 다음 계약은 유지한다.

1. 점수(사주 원본)와 해석(일기 근거)은 분리 표시
2. 상태 변경은 form POST + CSRF 검증
3. JS는 보조 역할만 수행(핵심 저장 로직은 서버)

## 11) 특정일 기준 Top5 기능 설계 (과거~미래)

### 목표
- 사용자가 특정 월(기준일)을 조회하면 아래를 함께 보여준다.
  - 과거 최고의 기회 Top5
  - 과거 최고의 위기 Top5
  - 미래 예상 기회 Top5
  - 미래 예상 위기 Top5

### 표시 원칙
- 과거 Top5는 **실기록 기반**으로 표시
- 미래 Top5는 **예측 기반**으로 표시
- 과거/미래를 같은 확실도로 보여주지 않고, 각 항목에 `신뢰도(%)`를 함께 표시

### 점수 산식(설계)
- 기본 강도 점수 `BaseScore` = 사주 원점수(job/gui/hlt 조합)
- 근거 보강 점수 `EvidenceScore` = 4행일기/이벤트 근거 일치도
- 최종 랭킹 점수 `RankScore` = `BaseScore + EvidenceScore`

주의:
- 일기/근거 데이터는 랭킹·해석에만 반영한다.
- 사주 원점수(job/gui/hlt) 자체는 절대 변경하지 않는다.

### 과거 Top5 선정 로직
1. 기준일 이전 구간에서 후보 월 집합을 만든다.
2. 각 월의 `RankScore`를 계산한다.
3. 기회는 높은 순, 위기는 낮은 순으로 정렬한다.
4. 상위 5개를 선택한다.
5. 각 항목에 `근거 요약(사주 지표 + 일기 키워드 + 사건 태그)`를 표시한다.

### 미래 Top5 선정 로직
1. 기준일 이후 구간에서 후보 월 집합을 만든다.
2. 각 월의 `BaseScore`를 계산한다.
3. 과거 유사 패턴(같은/유사 지표 구간)의 결과 분포를 사용해 보정값을 만든다.
4. `PredictedRankScore`로 정렬해 기회/위기 Top5를 만든다.
5. 각 항목에 `신뢰도(%)`와 `시나리오(낙관/기준/보수)`를 함께 표시한다.

### 신뢰도 산정(초기 규칙 기반)
- 신뢰도는 아래 3요소 가중 평균으로 계산한다.
  1. 데이터 충실도(해당 구간의 일기/이벤트 밀도)
  2. 패턴 일치도(과거 유사 시기와의 거리)
  3. 최근성 가중치(최근 데이터일수록 가중)
- 표시 예시: `신뢰도 78% (중)`

### 출력 형식(SSR 페이지)
- API 없이 `GET /`에서 서버가 계산 후 템플릿에 주입한다.
- 섹션 구성:
  - `기회 Top5 (과거)`
  - `위기 Top5 (과거)`
  - `기회 Top5 (미래예측)`
  - `위기 Top5 (미래예측)`
- 각 행: `월`, `강도점수`, `신뢰도`, `근거요약`

## 12) 테스트/검증 확장 (Top5 기능 포함)

1. 정확성 검증
- 과거 Top5는 실제 저장된 데이터만으로 재현 가능해야 한다.
- 같은 입력 데이터에서 같은 Top5 결과가 반복 재현되어야 한다.

2. 예측 검증
- 미래 Top5는 백테스트 구간에서 hit-rate(적중률)를 기록한다.
- 월별로 예측/실제 차이를 로그로 남긴다.

3. 안정성 검증
- 일기 입력 전후에 사주 원점수 불변성 검증
- 항목이 5개 미만인 경우(데이터 부족) 안전하게 축소 표시

4. 보안 검증
- Top5 근거요약 렌더링 시 escape 유지(XSS 차단)
- 기준일 파라미터 검증(포맷/범위)
